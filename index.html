AiGRI System Architecture

Complete Logic Flow for Agricultural Disaster Assessment

Phase 1: Pre-Disaster Setup

Registration & Baseline Creation

sequenceDiagram participant Farmer participant LGU_Operator as LGU Operator participant Frontend participant Backend as Backend (API) participant DB as Database participant AI_Pipeline as AI Pipeline (Async) participant File_Storage as File Storage %% Step 1: Farmer Registration & Plot Definition Farmer->>Frontend: 1. Registers account & draws farm boundary on map Frontend->>Backend: 2. POST /api/plots (farmer_info, boundary_geojson) Backend->>DB: 3. Create Farmer record & FarmPlot record DB-->>Backend: 4. Return new plot_id Backend-->>Frontend: 5. Confirm plot registration Frontend-->>Farmer: 6. Show "Plot Registered" confirmation %% Step 2: LGU Operator Initiates Baseline Mission LGU_Operator->>Frontend: 7. Views list of registered plots & initiates baseline mission Note right of LGU_Operator: Flies drone over the farm (offline action) LGU_Operator->>Frontend: 8. Uploads raw "baseline" drone images for the plot %% Step 3: Backend Processes Baseline Images Frontend->>Backend: 9. POST /api/missions/baseline (plot_id, images) Backend->>File_Storage: 10. Save raw baseline images to a unique mission folder Backend->>DB: 11. Create DroneMission record (status: 'processing_baseline') %% Step 4: Asynchronous AI Processing Backend->>AI_Pipeline: 12. Trigger background job: create_baseline(mission_id) Backend-->>Frontend: 13. Immediately respond with { "status": "Processing Started" } AI_Pipeline->>File_Storage: 14. Stitch images into baseline_map.tif AI_Pipeline->>File_Storage: 15. Save the generated orthomosaic map AI_Pipeline->>DB: 16. Update DroneMission record (status: 'baseline_complete', map_path)

Phase 1 Key Steps:

  • Farmer Onboarding (1-6): Account creation and farm boundary definition using GeoJSON
  • LGU Action (7-8): Drone flight and baseline image capture
  • Backend Ingestion (9-11): Secure file storage and mission record creation
  • AI Processing (12-16): Asynchronous image stitching and orthomosaic generation

Phase 2: Post-Disaster Analysis

Damage Assessment & Report Generation

sequenceDiagram participant LGU_Operator as LGU Operator participant Farmer participant Frontend participant Backend as Backend (API) participant DB as Database participant AI_Pipeline as AI Pipeline (Async) participant File_Storage as File Storage %% Step 1: LGU Operator Initiates Post-Disaster Mission Note over LGU_Operator: A typhoon/disaster occurs (external trigger) LGU_Operator->>Frontend: 1. Finds the existing mission (with completed baseline) Note right of LGU_Operator: Flies the exact same drone mission again (offline action) LGU_Operator->>Frontend: 2. Uploads raw "post-disaster" drone images %% Step 2: Backend Triggers the Full Analysis Frontend->>Backend: 3. POST /api/missions/analyze (mission_id, images) Backend->>File_Storage: 4. Save raw post-disaster images Backend->>DB: 5. Update DroneMission record (status: 'processing_analysis') %% Step 3: Asynchronous AI Processing Backend->>AI_Pipeline: 6. Trigger background job: run_damage_analysis(mission_id) Backend-->>Frontend: 7. Immediately respond with { "status": "Analysis in Progress" } Note over AI_Pipeline: The core AI workflow happens here AI_Pipeline->>File_Storage: 8. Stitch post-disaster images into post_disaster_map.tif AI_Pipeline->>DB: 9. Fetch plot boundary & baseline_map_path AI_Pipeline->>File_Storage: 10. Load both maps and clip them to the boundary AI_Pipeline->>AI_Pipeline: 11. Run analysis (e.g., U-Net/NDVI) to create a damage_mask AI_Pipeline->>AI_Pipeline: 12. Calculate damage % from the mask AI_Pipeline->>AI_Pipeline: 13. Generate secure, geotagged PDF Certificate AI_Pipeline->>File_Storage: 14. Save the generated PDF Certificate AI_Pipeline->>DB: 15. Create DamageReport record with all results & PDF path AI_Pipeline->>DB: 16. Update DroneMission record (status: 'complete') %% Step 4: Farmer/LGU Views the Final Report Farmer->>Frontend: 17. Logs in, sees mission is "Complete", and requests the report Frontend->>Backend: 18. GET /api/reports/{report_id} Backend->>DB: 19. Fetch report metadata (damage %, PDF URL, etc.) Backend-->>Frontend: 20. Return report data as JSON Frontend-->>Farmer: 21. Display results and provide a link to download the PDF certificate

Phase 2 Key Steps:

  • Mission Trigger (1-2): Post-disaster drone flight and image upload
  • Analysis Kick-off (3-5): Backend processes and stores new images
  • Core AI Pipeline (6-16): Image stitching, comparison, damage analysis, and PDF generation
  • Report Access (17-21): Farmer retrieves final damage assessment certificate

  • Asynchronous Processing: Heavy AI computations don't block user interactions
  • Two-Phase Design: Clear separation between baseline establishment and damage assessment
  • Secure File Management: Raw images and generated reports stored separately from database
  • Real-time Status Updates: Users can track mission progress without waiting
  • Tamper-proof Certificates: Generated PDFs serve as official damage documentation
